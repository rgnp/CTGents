# 📋 评审系统设计

---

## 文件结构

```
ideas/reviews/
├── README.md                         ← 本文件（系统说明）
├── pata/                             ← 每个 idea 一个子目录
│   ├── v1.md                         ← 第1轮评审
│   ├── v2.md                         ← 第2轮评审
│   └── ...
└── {other-idea}/
    └── v1.md
```

## 单个评审文件的格式

```markdown
# {Idea名} — 评审 v{n}

| 版本 | 角色 | 日期 |
|:----|:-----|:----|
| v{n} | 评审人: 冲塔怪 | 202X-XX-XX |
| v{n} | 回复: claude | 202X-XX-XX |

---

{冲塔怪写的内容，随便怎么写，零格式要求}

---

## 回复

{claude的回复，按实际情况写，通常包含}

### 同意并修改的
| 评审意见 | 修改内容 |
|:--------|:--------|
| xxx | xxx |

### 反驳
> 引用原评审

{反驳理由}

### 需讨论
> 问题

{回应}

---

## 决议

- [x] 修改方案后下一轮
- [ ] 继续讨论（仍有分歧，不修改先讨论清楚）
- [ ] 通过（可以开始实施）

{如果"修改方案后下一轮"，列一个todo清单}
- [ ] todo1
- [ ] todo2
```

---

## 评审流程

```
claude创建空评审文件 → 冲塔怪写评审 → claude读后回复+决议
                              ← 驳回/追加 ←  (决议=继续讨论)
                                         ↓
                              claude修改方案 → 创建下一轮评审文件
```

### 规则

| 规则 | 说明 |
|:----|:-----|
| **零格式要求** | 评审人随便写，想到什么写什么，不需要按维度填表 |
| **回复即决议** | 每次回复末尾标注决议状态，不拖 |
| **小改直接改** | 无争议的修改直接改方案，不改就记录在案 |
| **大改先讨论** | 如果我和评审人有根本分歧，先讨论清楚再改方案 |
| **一个文件=一轮** | 评审+回复都在同一个 v{n}.md 里，不拆文件 |
